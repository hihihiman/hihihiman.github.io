错题笔记

  • 保护期受时间限制：发表权
  • 权衡点：加密级别
  • 可用性：Ping/Echo
  • 性能：队列调度
  • 可修改性：信息隐藏
  • SoC 不是一块处理器芯片，而是芯片的集成
  • G2G：人口信息采集处理和利用业务
  • B2G：参加政府工程交接
  • 网闸：会组织学生用户通过校园网访问系统
  • RAID：3 块 80G 的 =》 80+80 = 160G；2 块 80G+1 块 40G =》 3 块 40G =》40+40 = 80G
  • 架构经典风格
    • 数据流：批处理、管道过滤器
    • 调用返回：主程序/子程序、面向对象、层次风格
    • 独立构建风格：进程通信风格、事件驱动风格
    • 虚拟机风格：解释器、基于规则的系统
    • 仓库风格：数据库风格、黑板风格、超文本系统

一、架构设计基础

1 计算机系统基础

  • 冯·诺伊曼计算机结构 = 运算器+控制器+存储器+输入设备+输出设备，中央处理器CPU = 运算器+控制器
  • DSP：Digital Signal Processor 信号处理器，专用于实时的数字信号处理，常采用哈佛体系结构
  • 操作系统具有并发性、共享性、虚拟性和不确定性
  • CORBA：Common Object Request Broker Architecture，公共对象请求代理架构
    • ORB：对象请求代理
    • 公共对象服务
    • 公共设施
  • EJB 是 J2EE的构件标准，其中的构件称为 Bean，可以分为会话Bean，实体Bean和消息驱动Bean
  • UML中四种关系
    • 依赖：使用的关系
    • 关联：拥有的关系，组合关系和聚合关系的区别在于组合关系中的“部分”只能属于一个“整体”，且生命周期相同
    • 泛化：父子继承关系
    • 实现：接口和实现之间
    • e.g. 在线学习系统中，课程学习和课程考试都需要先检查学员的权限，“课程学习”与“检查权限”两个用例之间属于包含关系。
  • UML中五种视图
    • 实现视图：开发者，查看配置
    • 进程视图：集成者，查看性能
    • 用例视图：测试者，查看行为
    • 部署视图：系统工程师，查看拓扑
    • 逻辑视图：最终用户，查看功能，描述涉及的对象模型和对象间的关系

2 嵌入式基础

  • 嵌入式系统具有操作系统的基本功能，如多任务调度、同步机制。
  • 嵌入式系统分层
    • 硬件层
    • 抽象层：硬件抽象层（HAL） 提供硬件资源，将操作系统与硬件平台隔开，使操作系统具有硬件无关性；板级支持包（BSP）提供对硬件的管理。
    • 操作系统层
    • 中间件层
    • 应用层
  • 存储器分类
    • RAM：Random Access Memory 随机存取存储器
      • DRAM：Dynsmic ～ 动态，常作主存
      • SRAM：Static ～ 静态，常作 Cache 缓存
    • ROM：Read Only Memory 只读存储器
      • MROM：Mask ～ 掩膜型，不可修改
      • PROM：Programmable ～ 可编程，一次性烧录
      • EPROM：Erasable ～ 可擦，通过紫外线擦写
      • EEPROM：Electronically ～ 电，通过电压清除
      • Flash Memory：快闪存储器，可以联机擦写
  • 存取速度排序（快->慢）：寄存器组>Cache 缓存>内存>Flash（ROM）
  • 嵌入式系统降低功耗一般采用：软硬件协同设计、编译优化、算法优化、“中断”代替“查询”、有效管理电源。

3 计算机网络基础

  • 复用技术：TDM 时分、FDM 频分、CDM 码分
  • OSI 七层
    • 物理层：传输比特流
      • 集线器（Hub）和中继器（Repeater）
    • 数据链路层：点到点的帧传输
      • 网桥（Bridge）和交换机（Switcher）
      • 在以太网标准中规定的最小帧长时 64B，这是根据网络中检测冲突的最长时间来确定的
    • 网络层：ip节点之间传输
      • 路由器（Router）和防火墙（Firewall）
    • 传输层：port端到端传输
    • 会话层：管理进程间通信
    • 表示层：数据编码
    • 应用层：处理网络应用，提供用户接口
  • FTP：File Transport Protocol，文件传输协议，基于TCP，21端口传送控制信息，20端口传送文件内容
  • TFTP：Trivial FTP，简单文件传输协议，基于UDP，69端口，不提供认证授权机制，采用超时重传来保证数据到达
  • HTTP：基于TCP，80端口
  • HTTPS：HTTP+SSL，SSL = Secure Socket Layer，安全套接层，使用443端口
  • DHCP：Dynamic Host Configuration Protocol 动态主机配置协议，如果网络中有多个DHCP服务器发送OFFER报文，客户端只根据第一个收到的OFFER报文，返回REQUEST报文
  • DNS：域名 -> IP地址；PTR：Pointer Record，IP地址 -> 域名
  • 网络设计分层：接入层（用户管理）、汇聚层（过滤）、核心层（双机冗余热备份）
    • 备用路径在主路径失效时才启用

背景 500

本人就职于杭州一家互联网公司，担任技术总监，主要负责参与一个短视频直播生态服务平台的设计开发和维护管理工作，项目一期的工期是六个月，涉及一位产品、一位前端、三位后端、一位测试、一位运维以及四位实习生。现如今，通过互联网平台带货已成为当前网络购物比较盛行的方式，这种新型的服务方式，给人们带来了很大的便利。各大直播平台诸如抖音、快手、淘宝等，都为播主提供了带货的直播通道，这不仅能够满足用户的购买需求，也能够增加商品的销量。在这一环节中，产品的分发和招商等渠道尚未完善。在传统的带货模式中，都是商家和主播之间进行商品交易，主播依靠卖出商品数量获取提成。在这种模式下，商家很难找到合适的带货主播，主播也很难找到合适的带货商品。短视频直播生态服务平台通过大数据追踪短视频内容生产及流量趋势，更好的助力账号内容定位、粉丝增长、电商带货及直播流量变现，为主播提供一个可供选择的、可靠的第三方交易平台，使得直播带货产生的效益最佳，帮助商家最大化提升销售转化。用户通过注册即可使用数据查询、平台交易等服务，还利用大数据云计算及算法建模技术，科学分析工商数据的内在价值，建立播主分析模型、视频分类模型、多维数据监控等多维度产品体系，使用户全方位了解播主、商品、视频、品牌及机构状况。短视频直播生态服务平台需要部署在公共服务器上。面向所有互联网上的合法用户，用户通过http方式于公网访问该服务器，可以调取系统服务。可由注册该平台账号的主播、品牌商自行查询相关数据并承接发布相关的任务,并且由系统管理员在执行时进行监管。
